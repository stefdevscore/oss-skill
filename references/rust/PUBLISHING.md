# crates.io Publishing Reference

## 1. Overview

This reference covers publishing Rust crates to [crates.io](https://crates.io) — from preparing `Cargo.toml` metadata through publishing, yanking, and release automation.

## 2. Build & Content (Cargo.toml Metadata)

```toml
[package]
name = "my-crate"
version = "1.0.0"
edition = "2024"
description = "A clear, one-line description (required)"
license = "MIT"
repository = "https://github.com/user/my-crate"
readme = "README.md"
keywords = ["keyword1", "keyword2"]           # max 5
categories = ["command-line-utilities"]        # max 5, from crates.io list
```

### Required vs Recommended Fields

| Field | Required | Notes |
|---|---|---|
| `name` | Yes | Must be unique on crates.io |
| `version` | Yes | SemVer |
| `description` | Yes | Plain text, shown in search |
| `license` | Yes | SPDX identifier (e.g., `MIT`, `MIT OR Apache-2.0`) |
| `edition` | Recommended | Defaults to 2015 if omitted |
| `repository` | Recommended | Link to source |
| `readme` | Recommended | Rendered on crates.io |
| `keywords` | Recommended | Up to 5, for search |
| `categories` | Recommended | Up to 5, from [crates.io/categories](https://crates.io/categories) |
| `documentation` | Optional | Defaults to docs.rs |
| `homepage` | Optional | Separate from repository |
| `rust-version` | Recommended | MSRV |

### Dual Licensing

The Rust ecosystem convention is dual licensing under MIT and Apache-2.0:

```toml
license = "MIT OR Apache-2.0"
```

This requires two license files:
```
LICENSE-MIT
LICENSE-APACHE
```

**Why dual license?** Apache-2.0 provides patent protection. MIT provides simplicity. Users choose whichever suits them. Most major Rust crates follow this convention.

### Package Contents

### What Gets Published

By default, everything in the repo is included except:

- Files in `.gitignore`
- `target/` directory
- Hidden files (dotfiles)

### Controlling Included Files

```toml
[package]
include = [
    "src/**/*",
    "Cargo.toml",
    "LICENSE*",
    "README.md",
    "CHANGELOG.md",
]
# OR
exclude = [
    "tests/",
    "benches/",
    "examples/",
    ".github/",
]
```

**Prefer `include`** — it's a more secure allowlist approach.

### Verifying Package Contents

```bash
# Preview what will be published
cargo package --list

# Create the .crate file without publishing
cargo package

# Inspect the package
ls -la target/package/

# Verify it builds from the packaged source
cargo package --verify
```

## 3. Authentication

### Recommended Default

Prefer non-interactive publishing in CI with a registry token or trusted publishing. Use `cargo login` mainly for local/manual publishing.

### Login

```bash
# Login with crates.io API token
cargo login

# Paste your token from https://crates.io/settings/tokens
```

The token is stored in `~/.cargo/credentials.toml`.

### Scoped Tokens

Create tokens with specific permissions at [crates.io/settings/tokens](https://crates.io/settings/tokens):

| Scope | Allows |
|---|---|
| **publish-new** | Publish new crates |
| **publish-update** | Publish new versions of existing crates |
| **yank** | Yank/unyank versions |

### CI Authentication

```yaml
# GitHub Actions
- run: cargo publish
  env:
    CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
```

If your registry supports trusted publishing for your workflow, that is also a good default because it avoids long-lived tokens.

## 4. Publishing Commands

### First-Time Publish

```bash
# Verify the package
cargo package --verify

# Dry run (checks everything but doesn't upload)
cargo publish --dry-run

# Publish
cargo publish
```

### Publishing Workspace Members

```bash
# Publish a specific crate in a workspace
cargo publish -p my-crate

# Order matters — publish dependencies first
cargo publish -p core-crate
cargo publish -p cli-crate   # depends on core-crate
```

### Automated Publishing with cargo-release

```bash
cargo install cargo-release

# Bump version, commit, tag, and publish
cargo release patch    # 1.0.0 → 1.0.1
cargo release minor    # 1.0.0 → 1.1.0
cargo release major    # 1.0.0 → 2.0.0

# Dry run
cargo release patch --dry-run
```

Configure in `Cargo.toml` or `release.toml`:

```toml
# release.toml
pre-release-commit-message = "chore: release {{version}}"
tag-message = "{{version}}"
tag-prefix = "v"
```

## CI/CD Release Workflow

```yaml
# .github/workflows/release.yml
name: Release
on:
  push:
    tags: ['v*']

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo publish
        env:
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
```

### Binary Releases with cargo-dist

For CLI tools, also distribute pre-built binaries:

```yaml
# Generated by cargo-dist
# Produces binaries for Linux, macOS, Windows
# Creates GitHub Release with installers
```

```bash
cargo install cargo-dist
cargo dist init
# Follow prompts, commit generated CI config
```

## 5. Versioning Semantics

### SemVer in Rust

Cargo enforces SemVer. The [SemVer compatibility page](https://doc.rust-lang.org/cargo/reference/semver.html) documents exactly what constitutes a breaking change in Rust:

**Breaking (major bump):**
- Removing a public item
- Changing a function signature
- Adding a required field to a public struct (without `#[non_exhaustive]`)
- Tightening generic bounds

**Non-breaking (minor bump):**
- Adding new public items
- Adding new optional features
- Deprecating items

**Patch:**
- Bug fixes
- Performance improvements
- Documentation updates

### `#[non_exhaustive]`

Use on public structs and enums to allow adding variants/fields without breaking changes:

```rust
#[non_exhaustive]
pub struct Config {
    pub name: String,
    pub verbose: bool,
    // can add fields in minor versions
}

#[non_exhaustive]
pub enum Error {
    NotFound,
    Timeout,
    // can add variants in minor versions
}
```

### Yanking

Yanking prevents new projects from depending on a version, but existing users are unaffected:

```bash
# Yank a version
cargo yank --version 1.0.1

# Unyank
cargo yank --version 1.0.1 --undo
```

- Yanked versions still work in existing `Cargo.lock` files.
- **Prefer yanking over any other removal** — crates.io does not support true unpublishing.

### Crate Naming

- Use lowercase with hyphens: `my-crate` (Cargo normalizes hyphens and underscores).
- The Rust import name uses underscores: `use my_crate::foo;`
- Check availability: search on [crates.io](https://crates.io).
- Name squatting is against crates.io policy.

### Pre-Publish Checklist

- [ ] `cargo build --release` succeeds
- [ ] `cargo test` passes
- [ ] `cargo clippy -- -D warnings` reports no issues
- [ ] `cargo fmt -- --check` passes
- [ ] `cargo doc --no-deps` builds cleanly
- [ ] `cargo package --verify` succeeds
- [ ] `cargo publish --dry-run` succeeds
- [ ] `Cargo.toml` has required fields (name, version, description, license)
- [ ] `keywords` and `categories` are set
- [ ] README contains installation and usage instructions
- [ ] CHANGELOG is updated
- [ ] `rust-version` (MSRV) is set and tested in CI
- [ ] Git tag matches the version
